Instead of polling `GET /v1/jobs/{job_id}` in a loop, you can register a webhook endpoint and let PureFrame push a notification to your server the moment a video finishes processing. This guide covers creating and managing webhook endpoints, understanding the event payload, and verifying your setup before going to production.

## Events

PureFrame currently emits the following event type:

| Event | When it fires |
| --- | --- |
| `video.processed` | A video has finished indexing (status `done` or `failed`) |

You subscribe to one or more event types when you create an endpoint. PureFrame will only deliver the events you've subscribed to.

---

## Creating a webhook endpoint

Register a URL on your server with `POST /v1/webhooks`. Supply the destination `url` and an array of `events` to subscribe to:

```bash
curl -X POST https://api.pureframe.ai/v1/webhooks \
  -H "Authorization: Bearer pf_your_key" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://yourapp.com/webhooks/pureframe",
    "events": ["video.processed"]
  }'
```

A `201 Created` response returns the new endpoint object:

```json
{
  "data": {
    "endpoint_id": "wh_jkl321",
    "url": "https://yourapp.com/webhooks/pureframe",
    "events": ["video.processed"],
    "is_active": true,
    "created_at": "2024-11-01T10:00:00Z"
  }
}
```

Save the `endpoint_id` — you'll use it to test, inspect, and manage this endpoint.

### Event payload

When PureFrame delivers an event, your server receives a `POST` with a JSON body like this:

```json
{
  "event": "video.processed",
  "endpoint_id": "wh_jkl321",
  "delivery_id": "dlv_mno654",
  "timestamp": "2024-11-01T10:06:22Z",
  "data": {
    "video_id": "vid_ghi789",
    "job_id": "job_def456",
    "collection_id": "col_abc123",
    "status": "done",
    "segments_count": 142,
    "duration_seconds": 284
  }
}
```

Respond with any `2xx` status code to acknowledge the delivery. If your server returns a non-2xx response or does not respond within 10 seconds, PureFrame marks the delivery as failed and retries with exponential backoff.

## Per-upload webhooks

You can also attach a one-time notification URL directly to an upload, without creating a permanent endpoint. Pass `webhook_url` as a form field in `POST /v1/upload`:

```bash
curl -X POST https://api.pureframe.ai/v1/upload \
  -H "Authorization: Bearer pf_your_key" \
  -F "collection_id=col_abc123" \
  -F "file=@/path/to/video.mp4" \
  -F "webhook_url=https://yourapp.com/webhooks/pureframe"
```

PureFrame delivers a `video.processed` event to this URL when the job completes. Unlike registered endpoints, per-upload webhook URLs are not stored and have no retry history.

## Testing an endpoint

Before deploying to production, send a synthetic test event to your endpoint with `POST /v1/webhooks/{endpoint_id}/test`:

```bash
curl -X POST https://api.pureframe.ai/v1/webhooks/wh_jkl321/test \
  -H "Authorization: Bearer pf_your_key"
```

PureFrame immediately dispatches a sample `video.processed` payload to your registered URL so you can verify that your server receives, parses, and acknowledges it correctly.

<Tip>
  Use the test endpoint before going to production. Confirm in your server logs that the payload arrives, your handler parses the `event` and `data` fields without errors, and you return a `200` response within the timeout window.
</Tip>

## Viewing delivery history

Inspect past delivery attempts for a registered endpoint with `GET /v1/webhooks/{endpoint_id}/deliveries`. Results are paginated with `page` and `per_page` query parameters:

```bash
curl "https://api.pureframe.ai/v1/webhooks/wh_jkl321/deliveries?page=1&per_page=20" \
  -H "Authorization: Bearer pf_your_key"
```

Each delivery record includes:

```json
{
  "data": {
    "deliveries": [
      {
        "delivery_id": "dlv_mno654",
        "event": "video.processed",
        "status": "delivered",
        "response_status_code": 200,
        "attempted_at": "2024-11-01T10:06:23Z",
        "duration_ms": 241
      },
      {
        "delivery_id": "dlv_pqr987",
        "event": "video.processed",
        "status": "failed",
        "response_status_code": 503,
        "attempted_at": "2024-11-01T10:08:10Z",
        "duration_ms": 10002
      }
    ],
    "page": 1,
    "per_page": 20,
    "total": 2
  }
}
```

| Status | Meaning |
| --- | --- |
| `delivered` | Your server returned a `2xx` response |
| `failed` | Your server returned a non-2xx response or timed out |
| `pending` | Scheduled for retry |

## Retrying a failed delivery

Manually trigger a retry for any failed delivery with `POST /v1/webhooks/{endpoint_id}/deliveries/{delivery_id}/retry`:

```bash
curl -X POST \
  https://api.pureframe.ai/v1/webhooks/wh_jkl321/deliveries/dlv_pqr987/retry \
  -H "Authorization: Bearer pf_your_key"
```

PureFrame immediately re-dispatches the original event payload to your endpoint URL. Use this after fixing a server-side issue that caused the original delivery to fail.

<Note>
  PureFrame also retries failed deliveries automatically with exponential backoff — up to 5 attempts over approximately 24 hours. Manual retries are available at any time, regardless of automatic retry status.
</Note>

## Updating an endpoint

Change the destination URL, modify subscribed events, or pause an endpoint with `PATCH /v1/webhooks/{endpoint_id}`:

<CodeGroup>

```bash Update URL
curl -X PATCH https://api.pureframe.ai/v1/webhooks/wh_jkl321 \
  -H "Authorization: Bearer pf_your_key" \
  -H "Content-Type: application/json" \
  -d '{"url": "https://yourapp.com/webhooks/pureframe-v2"}'
```

```bash Update subscribed events
curl -X PATCH https://api.pureframe.ai/v1/webhooks/wh_jkl321 \
  -H "Authorization: Bearer pf_your_key" \
  -H "Content-Type: application/json" \
  -d '{"events": ["video.processed"]}'
```

```bash Pause the endpoint
curl -X PATCH https://api.pureframe.ai/v1/webhooks/wh_jkl321 \
  -H "Authorization: Bearer pf_your_key" \
  -H "Content-Type: application/json" \
  -d '{"is_active": false}'
```

</CodeGroup>

Setting `is_active` to `false` pauses delivery without deleting the endpoint or its history. Set it back to `true` to resume.

---

## Deleting an endpoint

Permanently remove a webhook endpoint with `DELETE /v1/webhooks/{endpoint_id}`:

```bash
curl -X DELETE https://api.pureframe.ai/v1/webhooks/wh_jkl321 \
  -H "Authorization: Bearer pf_your_key"
```

A `204 No Content` response confirms the endpoint has been deleted. Delivery history for the endpoint is also removed.

<Warning>
  Deleting an endpoint is permanent. If you only need to stop deliveries temporarily, use `PATCH` to set `is_active: false` instead.
</Warning>

## API reference

- [POST /v1/webhooks](/api-reference/webhooks/create) — Create a webhook endpoint
- `GET /v1/webhooks/{endpoint_id}/deliveries` — List delivery history
- `POST /v1/webhooks/{endpoint_id}/test` — Send a test event
- `POST /v1/webhooks/{endpoint_id}/deliveries/{delivery_id}/retry` — Retry a failed delivery
- `PATCH /v1/webhooks/{endpoint_id}` — Update an endpoint
- `DELETE /v1/webhooks/{endpoint_id}` — Delete an endpoint
